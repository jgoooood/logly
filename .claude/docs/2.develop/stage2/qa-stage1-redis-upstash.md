## Q1. Rate Limit Redis 전환 < 이건 왜 필요한거야? 그리고 REDIS는 뭐야?

### Redis 가 뭐야?

**메모리에 데이터를 저장하는 초고속 키-값 데이터베이스**입니다.

| 일반 DB (Postgres) | Redis |
|---|---|
| 디스크에 저장 | 메모리(RAM)에 저장 |
| 복잡한 쿼리/관계 | 단순한 키-값(`set k v`, `get k`, `incr k`) |
| 응답 수~수백 ms | 응답 1ms 미만 |
| 용도: 영구 저장 | 용도: 캐시 / 카운터 / 세션 / 큐 |

**Upstash** 는 그 Redis 를 클라우드에서 호스팅해주는 서비스. 일반 Redis 는 서버 직접 운영이 필요한데, Upstash 는 HTTPS REST API 로 호출 (서버리스 환경인 Vercel 과 잘 맞음). 무료 티어 충분합니다.

---

### 왜 "전환"이 필요한가 — 현재 코드의 한계

지금(Round 1 종료 시점)의 `lib/rate-limit.ts` 는 이렇게 생겼습니다:

```ts
const buckets = new Map<string, Bucket>();   // ← 그냥 자바스크립트 Map
```

**즉, "Node.js 프로세스 메모리 안의 변수" 입니다.** 여기서 두 가지 운영 문제가 발생합니다.

#### 문제 1. 프로세스 재시작 시 카운터 초기화

악용 시나리오:
1. 공격자가 `/api/transform` 을 9회 호출 (일 한도 10)
2. 배포가 발생해서 서버 재시작 → `Map` 이 비어짐
3. 같은 사용자가 다시 9회 호출 가능 → 일 한도가 사실상 무한

Vercel 같은 서버리스 환경은 더 심각합니다. **요청 사이에 프로세스가 사라졌다 살아납니다.** 카운터가 거의 매번 초기화돼서 rate limit 자체가 무의미해집니다.

#### 문제 2. 멀티 인스턴스 동기화 안 됨

Vercel 운영 환경은 트래픽 따라 인스턴스 여러 개를 띄웁니다 (예: A, B 두 개).
- 사용자가 인스턴스 A 에 5회 호출 → A 의 Map: 5
- 같은 사용자가 인스턴스 B 에 5회 호출 → B 의 Map: 5
- 두 Map 은 서로 모름 → 사용자는 한도 10이 아니라 **각 인스턴스마다 10회씩** 사용 가능

#### 해법: "한 곳에 모아 두기"

A 도 B 도 모두 **같은 Redis** 한 곳을 보고 카운터를 더하면 위 문제가 모두 사라집니다. 그래서 외부 저장소가 필요하고, 빠르고 단순한 카운터 용도에는 Redis 가 정석입니다.

---

### 왜 지금 단계에서 하는가

CLAUDE.md §9.5 에 못 박혀 있습니다:

> `lib/rate-limit.ts` 는 Redis(Upstash) 또는 Postgres 카운터로 구현. **메모리 카운터 금지.**

Round 1 의 `Map` 구현은 "MVP 동안만 임시" 였고, Stage 8(운영 배포)의 의존으로 1단계가 잡혀 있는 이유입니다 — **운영에 인메모리 rate limit 으로 나가면 사실상 rate limit 이 없는 것과 같음.**

---

### 비유로 다시

- **인메모리 Map** = 가게 직원이 손님 머릿속으로 "이 사람 오늘 몇 번 왔지" 외우는 것. 직원이 교대하거나(=재시작) 매장이 여러 개면(=멀티 인스턴스) 무용지물.
- **Redis** = 매장들이 공유하는 클라우드 출입 명부. 어느 매장 어느 직원이 봐도 같은 카운터.

## Q2. 그럼 COUNT를 REDIS에 저장해서 사용자별 카운트를 절대적으로 관리하겠다는거지?

네, 정확합니다.

```
사용자 u123 이 변환 요청
  → Redis 에 "transform:u123" 키를 INCR (1 증가)
  → 결과 카운트가 limit(10) 이하면 통과, 넘으면 429
  → 첫 증가 시 EXPIRE 86400 (24시간) 걸어두면 하루 후 자동 리셋
```

### 핵심 속성 3가지

1. **단일 출처(single source of truth)** — Vercel 인스턴스가 A·B·C 여러 개여도, 모두 같은 Upstash Redis 한 곳을 본다. "사용자 u123 의 오늘 카운트" 는 우주에 단 하나만 존재.

2. **원자성(atomicity)** — Redis 의 `INCR` 은 원자 연산. 두 인스턴스가 동시에 증가시켜도 "둘 다 5번째" 같은 race 가 발생하지 않음. 9 → 10 → 11 순서대로 깔끔하게 증가.

3. **TTL 자동 만료** — 첫 증가 때 `EXPIRE 86400` 걸어두면 24시간 뒤 키가 자동 삭제 → 카운터가 0 으로 리셋. 별도 cron 이 필요 없음.

이렇게 해서 **"사용자별 일 10회"** 라는 비즈니스 규칙을 인스턴스 수·재시작·시간대와 무관하게 **절대적으로** 보장합니다. 우리가 작성한 코드(`lib/rate-limit.ts`) 도 정확히 이 흐름 그대로입니다 — `incr` → 1이면 `expire` → 한도 초과면 `ttl` 로 retryAfter 반환.

## Q3. VERCEL이 인스턴스가 2개로 분산되어도 하나의 REDIS를 보는거야?

네, 그게 핵심입니다.

```
        [사용자 요청]
              │
        ┌─────┴─────┐
        ▼           ▼
   Vercel A    Vercel B    ← Vercel 이 자동 분산 (어느 인스턴스로 갈지 모름)
        │           │
        └─────┬─────┘
              ▼
     ┌────────────────┐
     │  Upstash Redis │  ← 우리가 만든 단 하나의 DB
     │   (도쿄 리전)  │
     └────────────────┘
```

### 왜 "하나" 인가

Upstash 에서 Redis DB 를 만들면 **고유한 URL 한 개**가 발급됩니다.

```
UPSTASH_REDIS_REST_URL=https://apn1-xxxxx-12345.upstash.io
```

이 URL 을 Vercel 의 환경변수에 등록하면, **모든 인스턴스가 똑같은 환경변수**를 읽습니다. 즉:

- 인스턴스 A 의 코드 → `process.env.UPSTASH_REDIS_REST_URL` → `https://apn1-xxxxx-12345.upstash.io`
- 인스턴스 B 의 코드 → `process.env.UPSTASH_REDIS_REST_URL` → `https://apn1-xxxxx-12345.upstash.io`

→ 같은 주소 → 같은 Redis → 같은 카운터.

### 인스턴스가 늘어나도 동작

Vercel 이 트래픽 폭증으로 인스턴스를 10 개로 늘려도:

```
A B C D E F G H I J  (10 개)
└─┴─┴─┴─┴─┴─┴─┴─┴─┘
       │
       ▼
  Upstash Redis  ← 여전히 한 곳
```

10 개 인스턴스가 동시에 `INCR transform:u123` 을 호출해도 Redis 가 원자적으로 처리:
- A: INCR → 1 반환
- B: INCR → 2 반환
- C: INCR → 3 반환
- ... 순서가 어떻게 섞이든 카운터는 정확히 1, 2, 3, ... 으로 증가

겹치거나 빠지지 않음.

### 한 줄 정리

**"인스턴스는 일회용 워커 / Redis 는 공용 장부"** — Vercel 인스턴스가 몇 개로 늘어나든 줄어들든, 모두 동일한 Redis 한 곳을 보기 때문에 카운터는 항상 "사용자 1명당 1개" 로 유지됩니다.

이게 바로 인메모리 `Map` 으로는 불가능했던 부분입니다 (각 인스턴스마다 자기 Map 을 따로 가지고 있어서 동기화 안 됨).

## Q4. UPSTASH는 뭐야? 그리고 REDIS랑 어떻게 연계되는거야?

### 한 줄 요약

**Redis = 소프트웨어**, **Upstash = 그 Redis 를 클라우드에서 대신 운영해주는 회사.**

마치 "Postgres(DB) vs Supabase(Postgres 호스팅 회사)" 의 관계와 똑같습니다.

```
PostgreSQL  ←→  Supabase
   Redis    ←→  Upstash
```

---

### Redis 만 있으면 안 되나?

원칙적으로는 가능합니다. 하지만 직접 운영하려면:

| 직접 운영 시 해야 하는 일 | Upstash 사용 시 |
|---|---|
| 서버 임대 (AWS EC2 등) | 불필요 |
| Redis 설치 / 설정 / 보안 패치 | 불필요 |
| 백업 / 모니터링 / 장애 대응 | 불필요 |
| 트래픽 늘면 서버 증설 | 자동 |
| 24시간 켜둬야 → 비용 항상 발생 | **요청당 과금** |

특히 마지막 **"요청당 과금"** 이 우리 같은 작은 서비스에 결정적입니다.

---

### Upstash 의 특별한 점 — "서버리스 친화적"

일반 Redis 는 TCP 연결이 필요합니다. 그런데 Vercel 같은 서버리스 환경은:
- 요청마다 새 인스턴스가 떠서
- TCP 연결 맺고
- 응답 보내고
- 인스턴스가 사라짐

→ TCP 연결을 매번 새로 맺으면 느리고 비효율.

**Upstash 는 HTTP(S) REST API 로 Redis 를 제공**합니다:

```
일반 Redis:    클라이언트 ──TCP──▶ Redis (연결 유지 필요)
Upstash:       클라이언트 ──HTTPS─▶ Redis (요청마다 끊고 맺음, OK)
```

서버리스 환경에서 다른 API 호출하듯이 사용 가능. 이게 우리가 Upstash 를 고른 핵심 이유입니다.

---

### 연계 흐름 (실제 코드 기준)

#### 1. Upstash 콘솔에서 DB 생성

```
Upstash 콘솔 → "Create Database" → 리전 선택 (도쿄)
```

생성하면 두 가지 정보가 나옵니다:

```
REST URL:   https://apn1-xxxxx-12345.upstash.io
REST Token: AX...(긴 토큰)
```

#### 2. `.env.local` 에 등록 (Vercel 운영 환경에선 Vercel 환경변수)

```
UPSTASH_REDIS_REST_URL=https://apn1-xxxxx-12345.upstash.io
UPSTASH_REDIS_REST_TOKEN=AX...
```

#### 3. `@upstash/redis` 라이브러리가 SDK 역할

방금 우리가 설치한 패키지입니다. `lib/rate-limit.ts:23` 에서:

```ts
client = new Redis({ url, token });   // ← URL/토큰만 넘기면 끝
```

이 한 줄이 "Upstash 의 그 DB 와 통신할 클라이언트" 를 만듭니다.

#### 4. 호출

```ts
await redis.incr("transform:u123");   // ← 평범한 함수 호출처럼 보이지만
```

내부적으로는:

```
HTTPS POST https://apn1-xxxxx-12345.upstash.io
Authorization: Bearer AX...
Body: ["INCR", "transform:u123"]
   ↓
Response: { result: 1 }
```

→ Upstash 서버에서 진짜 Redis 명령(`INCR`)을 실행 → 결과를 HTTPS 응답으로 돌려줌.

---

### 비유로 정리

- **Redis** = "장부" (소프트웨어)
- **Upstash** = "장부를 대신 보관해주고 24시간 응대해주는 보관소" (회사)
- **REST URL/Token** = "보관소 주소 + 출입증"
- **`@upstash/redis` 라이브러리** = "보관소에 전화해서 장부 조회/수정 요청하는 전화기"
- **`redis.incr(...)`** = "전화 한 통: '내 장부 카운터 1 올려주세요'"

우리 코드는 Redis 가 어디서 어떻게 돌아가는지 신경 쓰지 않고, "Upstash 의 그 DB 에다 명령을 보낸다" 는 추상화 안에서만 작업합니다. **DB 본체의 운영(설치/백업/장애 대응)은 전부 Upstash 가 책임집니다.**

## Q5. UPSTASH를 무료로 사용하는거야?

네, 무료 티어로 충분합니다 — 적어도 MVP 단계에서는.

### Upstash 무료 티어 (대략, 2025~2026 기준)

| 항목 | 무료 한도 |
|---|---|
| 일일 명령 수 | **10,000 commands/day** |
| 저장 용량 | 256 MB |
| DB 개수 | 1 개 (특정 리전 1곳) |
| 대역폭 | 50 GB/월 |
| 신용카드 등록 | **불필요** |

> ※ 정책은 가끔 바뀌므로 실제 가입 시 콘솔 화면을 한 번 확인해보세요.

---

### 우리 서비스가 무료 티어 안에 들어오는가

Logly 의 Redis 사용량을 거칠게 계산해봅니다.

**1회 변환 요청당 Redis 명령 수**
- `INCR` — 1회
- `EXPIRE` — 첫 호출에만 1회 (그 외는 0)
- `TTL` — 한도 초과 시에만 1회 (대부분 0)

→ **평균 약 1~1.2 명령 / 요청**

**하루 10,000 명령 = 하루 약 8,000~10,000 회 변환 요청 가능**

| 사용자 수 | 사용자당 일 변환 | 일 명령 수 | 무료 티어 안? |
|---|---|---|---|
| 10명 | 10회 (한도 풀로) | ~120 | ✅ 여유 |
| 100명 | 5회 평균 | ~600 | ✅ 여유 |
| 1,000명 | 3회 평균 | ~3,600 | ✅ 여유 |
| 10,000명 | 3회 평균 | ~36,000 | ❌ 초과 |

→ 사용자가 **수천 명** 수준이 될 때까지 무료 티어로 충분합니다. 그쯤 가면 매출도 따라오니 유료 전환은 자연스러운 시점입니다.

### 유료 전환은 언제?

대시보드에 일일/월간 명령 수 그래프가 나옵니다. 80% 근처 가면 알람이 옵니다.

유료(`Pay as you go`):
- 신용카드 등록
- 명령 100,000 건당 약 $0.20 (저렴)
- 무료 한도까지는 여전히 무료, 초과분만 과금
